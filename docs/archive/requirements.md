# ChatImage 需求文档

## 1. 产品概述

ChatImage 是一个面向普通 ChatGPT 用户的网页应用。它将 LLM 原本以长文本形式输出的回答，转换成一张可点击交互的结构化图片，帮助用户更快抓住关键信息，并通过点击图片中的不同区域查看更详细的解释。

产品目标不是简单“把文字生成一张图”，而是把一次 LLM 回答变成一种新的阅读和理解界面：

- 第一层：图片中呈现标题、关键观点、结构关系、图标、分区、流程或对比信息。
- 第二层：用户点击图片中的热点区域后，查看该区域对应的详细文本。
- 第三层：用户可以围绕某个热点区域继续追问，形成局部对话分支。
- 第四层：系统保存 ChatImage 结果和每个热点的追问历史，用户刷新后仍可查看历史记录。
- 第五层：用户可以放大图片、保存图片，便于复看、分享或归档。

## 2. 背景与问题

当前用户与 LLM 交流时，默认输出形态主要是文本。对于简单问答，文本足够高效；但当回答内容较长、结构复杂、包含多步骤分析或多维度比较时，用户常遇到以下问题：

- 很难快速识别核心结论。
- 长段落阅读负担高，容易遗漏重点。
- 信息之间的层级、因果、对比和流程关系不够直观。
- 用户想深入某个局部时，需要在长文本中反复定位。

ChatImage 通过“结构化图片 + 点击详情”的方式，让用户先看全局，再按需展开局部细节。

## 3. 目标用户

MVP 目标用户为普通 ChatGPT 用户，包括：

- 日常使用 LLM 获取解释、建议、总结、计划的用户。
- 需要快速理解复杂内容的学生、职场用户、内容创作者。
- 对纯文本长回答阅读成本较敏感的轻量用户。

后续可扩展到知识管理、教育学习、报告生成、会议纪要、研究辅助等场景。

## 4. 用户价值

ChatImage 为用户提供的核心价值：

- 快速理解：用视觉结构呈现原回答的关键内容。
- 降低阅读负担：把长文本拆成可扫读的视觉模块。
- 按需深入：点击局部区域查看完整解释，而不是一次性阅读全部细节。
- 便于复用：支持放大查看和保存图片。
- 保留原意：图片内容和详情文本均来自原始 LLM 回答，减少信息丢失。

## 5. 核心使用流程

1. 用户在网页端输入问题。
2. 系统调用 LLM 生成原始文本回答，并保存该回答。
3. ChatImage 对原始回答进行结构化解析，提取标题、摘要、关键模块、模块详情、关系类型等信息。
4. ChatImage 基于结构化结果生成适合生图 API 的风格化提示词。
5. 第三方生图 API 生成包含标题、文本、图标、网格区域等视觉元素的图片。
6. 真实 API 模式下，ChatImage 将图片和模块列表交给独立视觉模型定位每个模块卡片的真实位置；普通文本模型不能承担该步骤。
7. ChatImage 根据视觉模型返回的实际 bounds 叠加透明热点层；mock 模式继续使用确定性布局坐标。
8. 用户看到可交互图片。
9. 用户点击图片中的区域，弹出该区域对应的详细信息。
10. 用户阅读详情后，可以围绕该区域继续追问，形成一个基于该热点区域的对话分支。
11. 用户可点击图片右下角外侧的放大镜查看大图，或保存图片到本地。

## 6. MVP 功能范围

### 6.1 聊天输入与回答生成

- 用户可以在网页端输入一个问题。
- 系统展示基本生成状态，例如“正在生成回答”“正在生成图片”。
- 系统保存本次 LLM 的原始文本回答。
- MVP 可以暂不展示完整文本回答，但应支持调试或后台查看，便于验证图片是否忠实。

### 6.2 文本结构化

系统需要将原始回答转换成结构化数据，至少包含：

- 总标题。
- 一句话摘要。
- 3 到 6 个主要信息模块。
- 每个模块的短标题。
- 每个模块的图片内短文本。
- 每个模块的详细说明文本。
- 每个模块对应的图标语义或视觉建议。
- 模块之间的关系类型，例如并列、流程、对比、层级。

### 6.3 图片生成

系统需要基于结构化数据调用第三方文本转图片 API，生成一张信息图风格图片。

图片应包含：

- 清晰标题。
- 与内容结构匹配的分区布局，例如网格、流程、对比、中心辐射、时间线或矩阵。
- 每个模块对应的短标题和短文本。
- 简洁 icon 或视觉符号。
- 足够清晰的视觉层级。
- 适合网页端展示的尺寸比例。

MVP 推荐图片比例：

- 默认：16:9，用于桌面端阅读。
- 可选：4:5 或 1:1，用于移动端或分享。

### 6.4 透明热点层

系统在图片上叠加透明点击区域。

每个热点区域需要：

- 覆盖图片中一个信息模块。
- 真实 API 模式下优先来自视觉对齐返回的真实卡片 bounds，而不是生图前的近似 LayoutSpec 坐标。
- 支持 hover 或点击反馈。
- 默认不覆盖图片内容；点击反馈应短暂出现并自动消失，避免常驻高亮遮挡图片。
- 点击后打开详情弹窗、侧边栏或浮层。
- 关联对应模块的详细说明文本。

透明热点层不是图片本身的一部分，而是前端组件在图片上叠加的交互层。

### 6.5 详情展示

点击热点区域后，展示该区域的详细内容，并允许用户围绕该区域继续追问。

MVP 详情内容包括：

- 模块标题。
- 模块详细说明。
- 可选的原文引用或来源片段。
- 该区域的追问输入框。
- 该区域的追问历史。

详情展示形态建议：

- 桌面端：右侧详情面板优先，避免追问时频繁打开和关闭弹窗。
- 移动端：底部抽屉。

### 6.6 区域追问与对话分支

ChatImage 需要支持类似 ChatGPT 的多轮追问能力。区别在于，用户的追问不是只基于整段原始回答，而是可以从图片中的某个热点区域发起。

用户体验要求：

- 用户点击热点区域后，右侧详情面板展示该区域详情和追问输入框。
- 用户在该区域内提问时，系统应理解当前追问的上下文是“原始问题 + 原始回答 + 当前热点区域 + 当前区域历史追问”。
- 每个热点区域拥有独立的对话分支。
- 用户切换到另一个热点区域时，应看到另一个区域自己的详情和追问历史。
- 用户可以回到之前点击过的热点区域，继续原来的追问分支。
- 区域追问的回答以文本形式展示在详情面板中，MVP 不要求每次追问都重新生成图片。

对话分支的边界：

- 主问题生成一张 ChatImage。
- 一张 ChatImage 下包含多个 hotspot。
- 每个 hotspot 可以开启一个 thread。
- thread 内支持多轮 user/assistant 消息。
- thread 回答应优先围绕该 hotspot 的 detail 展开，必要时引用原始回答的其他部分。

MVP 暂不要求用户从某个分支继续生成新的 ChatImage。后续可以支持“基于当前分支生成新图”，形成图像化的深挖路径。

### 6.7 放大与保存

- 图片右下角外侧显示放大镜按钮。
- 点击后进入大图查看模式。
- 大图模式支持关闭。
- 支持保存图片到本地。

MVP 保存范围可以先只保存静态图片；后续再支持保存交互版链接或 HTML 包。

### 6.8 历史记录与持久化

- 系统需要保存每次生成的原始问题、原始回答、结构化布局、图片地址、热点数据和创建时间。
- 系统需要保存每个 hotspot 的独立 thread 和消息列表。
- 刷新页面后，用户应至少能看到最近生成记录。
- MVP 可使用本地 SQLite；后续可迁移到 PostgreSQL 或云数据库。
- 不允许在服务端硬编码某个具体人物、机构或时效事实作为“已核验信息”。事实核验应通过独立检索或可信数据源模块实现。

## 7. 非 MVP 范围

以下能力不纳入第一版：

- 多轮上下文中的多张 ChatImage 自动管理。
- 浏览器插件接管 ChatGPT 官方网页。
- 用户自定义视觉模板市场。
- 团队协作、云端知识库、权限管理。
- 对图片内容进行复杂 OCR 校验。
- 离线运行。
- 交互式图片导出为可离线点击的文件包。

## 8. 关键产品约束

### 8.1 信息保真

ChatImage 生成的图片不能改变原始 LLM 回答的核心含义。结构化摘要可以压缩信息，但不应新增未经原回答支持的事实。

### 8.2 视觉容量

一张图片容量有限，图片内只展示短文本和关键结构；长解释应放在点击详情里。

### 8.3 热点准确性

透明热点层必须和图片中的视觉区域大致一致。若点击区域偏移严重，用户会认为产品不可靠。

布局可以根据内容自适应变化，但所有可点击热点必须来自同一份结构化布局数据。系统不能在图片生成完成后仅凭人工规则或前端猜测来决定点击区域。

### 8.4 生成耗时

一次完整生成需要经历 LLM 回答、结构化解析、生图、渲染等步骤。MVP 应尽量通过进度状态降低等待焦虑。

### 8.5 分支上下文清晰

区域追问必须让用户明确知道自己正在追问哪一个图片区域。系统需要避免把不同热点区域的追问历史混在一起，否则会破坏用户对图片结构的理解。

## 9. 用户体验要求

- 首页即聊天工作台，不做营销型落地页。
- 生成过程中展示清晰状态。
- 图片加载失败时，应允许用户重试生图。
- 若生图失败，应保留原始文本回答和结构化结果。
- 点击热点时反馈要明确，避免用户不知道图片可交互。
- 热点常态必须完全透明，不叠加文字、颜色块或常驻描边。
- 详情面板中需要明确显示当前选中的区域标题。
- 区域追问输入框应靠近当前区域详情，避免用户误以为是在追问整张图。
- 切换热点区域时，应保留每个区域已有的追问历史。
- 放大镜按钮应位于图片右下角外侧，避免遮挡图片内容。
- 移动端需要保证图片可缩放查看，详情内容可读。

## 10. 成功指标

MVP 阶段建议观察：

- 图片生成成功率。
- 从用户提问到图片可交互的总耗时。
- 用户点击热点区域的比例。
- 每张图片平均热点点击数。
- 有多少热点区域触发了追问。
- 每个对话分支的平均追问轮数。
- 用户切换热点后继续追问的比例。
- 用户是否保存图片。
- 用户是否选择重新生成图片。
- 用户对“是否比文本更容易理解”的主观反馈。

## 11. 主要风险

- 生图 API 对精确布局和文字渲染的控制不足。
- 图片中文字可能出现错字、乱码或语义偏差。
- 热点层坐标与图片视觉区域可能不匹配。
- 生成链路较长，响应速度可能慢。
- 区域追问的上下文边界不清，可能让回答偏离用户点击的局部。
- 多个热点区域都有追问历史时，前端状态和后端 thread 管理容易复杂。
- 普通用户可能不理解图片可点击，需要交互暗示。
- 成本可能高于纯文本聊天。

## 12. 待确认问题

- MVP 使用哪一家 LLM API 和生图 API。
- 是否需要展示原始文本回答。
- 是否允许用户选择图片风格。
- 图片默认比例选择 16:9、4:5 还是自适应。
- 详情和追问展示使用侧边栏、弹窗还是底部抽屉。
- 区域追问是否需要支持引用原文片段。
- 区域追问是否需要支持“生成新的 ChatImage”。
- 图片保存是否只保存静态图，还是需要保存交互版。
